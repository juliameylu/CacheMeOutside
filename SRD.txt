1. Introduction

1.1 Purpose

This Software Requirements Document (SRD) defines the current requirements for
HelpQ, a real-time office-hours queue management web application. The system
helps instructors and TAs manage high-volume help sessions while giving students
clear visibility into their wait position and queue progress.

1.2 Scope

Included in scope:

- Session creation and management for hosts, including instructors and TAs
- Real-time student queue joining and status tracking
- Student queue entries with name and question details
- Host queue controls for starting, completing, and removing entries
- Basic queue metrics, including active queue size and wait estimates
- Clear responsive interfaces for students, hosts, and first-time users

Out of scope for the current class project:

- University SSO or LMS integration
- Video call or chat tutoring features
- Native mobile apps
- Advanced analytics dashboards beyond core sprint goals

1.3 Intended Audience

This document is intended for:

- CSC 307 instructors and graders
- Student development team members
- Project stakeholders and peer reviewers

2. Product Vision

2.1 Vision Statement

For professors, TAs, and students who struggle with chaotic high-volume
office-hours sessions, HelpQ is a real-time office-hours queue management web
application which structures help requests, gives students live position
visibility, and lets hosts process queues efficiently. Unlike informal sign-up
sheets or ad hoc chat-based Q&A, our product provides a synchronized, role-based
queue workflow with real-time updates and basic anti-spam controls.

3. Overall Description

3.1 Product Overview

HelpQ is a collaborative office-hours queue tool for courses with many students
seeking help at once. Students join a session queue with their question, and
hosts process requests in an organized real-time workflow.

3.2 Product Perspective

The product is a standalone web application designed as a practical alternative
to informal office-hours management methods such as paper sign-ups, verbal
lines, or scattered messages. In this iteration, it focuses on fast queue
operations and visibility for both student and host roles.

3.3 Product Features

- Host-created office-hours sessions with a join code or link
- Real-time queue with live position updates
- Student question submission for each queue entry
- Host queue controls to serve, complete, or remove entries
- Basic queue status and wait information

3.4 User Classes and Characteristics

| User Class          | Characteristics                                          | Primary Goals                                                            |
| ------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------ |
| Student             | Joins sessions quickly and wants visibility and fairness | Enter queue fast, track position, and know when their turn is near       |
| Host (TA/Professor) | Runs office hours under time pressure                    | Create/manage sessions, process queue efficiently, and reduce disruption |
| First-Time User     | New to the queue workflow                                | Understand how to join/create sessions and submit clear help requests    |

3.5 Operating Environment

| Environment Area | Requirement                                                      |
| ---------------- | ---------------------------------------------------------------- |
| Hardware         | Any modern laptop or smartphone with internet access             |
| Software         | Modern web browser such as Chrome, Firefox, Safari, or Edge      |
| Network          | Stable internet connection for real-time queue synchronization   |
| Planned stack    | React, Node.js, Express, PostgreSQL, Socket.IO, and Tailwind CSS |

4. User Stories

4.1 Story Quality and INVEST Notes

Each story follows the required format: "As a (type of user), I want (desired
behavior/action) so that (benefit/justification/value)." The stories are written
to follow INVEST:

- Independent: each story can be discussed and planned separately.
- Negotiable: each story describes user value without locking the team into one
  implementation.
- Valuable: each story names a clear benefit for the user or team.
- Estimable: each story is small enough for the team to estimate during sprint
  planning.
- Small: each story focuses on one workflow or user need.
- Testable: each story can be verified through UI behavior, API behavior, or
  acceptance checks.

5. Functional Requirements

5.1 Session Creation and Join

- The system shall allow a host to create an office-hours session.
- The system shall generate a shareable join code or link for each active
  session.
- The system shall allow a student to enter a valid join code or open a valid
  join link.
- The system shall reject invalid or closed session codes with a clear error
  message.

5.2 Queue Entry and Live Position Updates

- The system shall allow a student to submit a queue entry with name and
  question summary.
- The system shall validate required queue-entry fields before saving the entry.
- The system shall add each new valid queue entry to the end of the session
  queue.
- The system shall persist queue entries in PostgreSQL.
- The system shall show students their current queue position and request
  status.
- The system shall broadcast queue updates to connected clients using Socket.IO.

5.3 Host Queue Moderation and Completion

- The system shall show the host an ordered queue with student names and
  question summaries.
- The system shall allow the host to mark an entry as in-progress.
- The system shall allow the host to mark an entry as complete.
- The system shall allow the host to remove duplicate, invalid, or spam entries.
- The system shall update all connected clients after host queue changes.

5.4 User Interface Requirements

- The student interface shall include a join/session entry screen.
- The student interface shall include a queue-entry form with clear validation
  messages.
- The student interface shall include a queue status view showing position and
  current status.
- The host interface shall include a session setup flow.
- The host interface shall include a queue dashboard with controls for entry
  status and removal.
- The application shall be usable on desktop and mobile browsers.

5.5 Data Requirements

The system shall store:

- Session ID
- Session join code or link token
- Session title or label
- Host identifier
- Student display name
- Student question summary
- Queue entry status
- Queue order/position
- Created and updated timestamps

6. Non-Functional Requirements

6.1 Performance Requirements

- 95% of API requests should respond within 500 ms under normal class-project
  load.
- The system should support at least 50 concurrent active users during testing.
- The system should show informative errors if a temporary performance or
  network issue occurs.

6.2 Security Requirements

- Users must log in before accessing host session and queue management features.
- Users may only view and modify queue data for sessions they are permitted to
  join or host.
- The deployment should use HTTPS.
- The system shall not store plaintext credentials in the repository.

6.3 Reliability Requirements

- The system should remain available during demo and grading windows.
- The team should restore a demo-blocking outage within 2 hours when reasonably
  possible.
- The system should avoid data loss during normal queue operations.

6.4 Usability Requirements

- New users should understand the core join or create-session workflow within 5
  minutes.
- The interface should use clear labels, readable contrast, and
  keyboard-accessible controls.
- The README should provide concise usage steps and troubleshooting notes.

6.5 Portability and Compatibility

- The application should support the latest two major versions of Chrome,
  Firefox, Safari, and Edge.
- The application should use a responsive layout for modern iOS and Android
  browsers.
- Development should work on macOS, Windows, and Linux when dependencies are
  installed.

6.6 Maintainability and Supportability

- The codebase shall use linting, formatting, and pull request review before
  merge when practical.
- The SRD and README shall be updated when major requirements or setup steps
  change.
- The team should respond to critical demo-blocking issues within 24 hours.

7. Use Cases

7.1 Use Case 1: Host Creates a Session

| Element        | Description                                                                                                         |

8. System Constraints, Assumptions, and Dependencies

8.1 Technical Constraints

- The project must be feasible for a student team to implement within one
  academic term.
- The project uses a web-first architecture.
- The planned stack is React, Node.js, Express, PostgreSQL, Socket.IO, and
  Tailwind CSS.
- Real-time features depend on stable client-server connectivity.

8.2 Business and Course Constraints

- The team must follow CSC 307 project expectations and deadlines.
- Scope should stay small enough to build and demonstrate during the term.
- Requirements may be adjusted after instructor feedback.

8.3 Assumptions

- Users will usually have a stable internet connection during office hours.
- Hosts will share the correct session code or link with students.
- Students will provide enough information in their question summary for hosts
  to prepare.

8.4 Dependencies

- React and browser APIs for the frontend user experience
- Express and Node.js for backend APIs
- PostgreSQL for persistent session and queue data
- Socket.IO for real-time updates
- Team availability for implementation, review, and testing

9. Acceptance Criteria

9.1 Functional Acceptance

- All core session creation, queue entry, queue status, and host management
  requirements are implemented and tested.
- Students can join a valid session and submit a queue entry without assistance.
- Hosts can view, advance, complete, and remove queue entries.
- Users receive clear feedback for invalid session codes or incomplete forms.

9.2 Non-Functional Acceptance

- The application remains usable during normal class-project testing.
- Basic security expectations are met for host-only actions and credential
  handling.
- The interface is readable, responsive, and accessible for common desktop and
  mobile browsers.
- Documentation explains how to run and test the project locally.

9.3 Quality Acceptance

- Code passes formatting and linting checks where configured.
- No critical bugs remain before demo submission.
- The SRD, README, and issue board reflect the current project scope.
